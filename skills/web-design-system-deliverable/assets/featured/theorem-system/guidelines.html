Design System / Guidelines

How We Build

Principles, patterns, and production rules that govern every surface of Theorem. This is not a suggestion list. These guidelines are load-bearing.

In this document

01

Voice & Tone

Theorem speaks like a senior engineer with design taste. Every word should earn its place. We are precise without being cold, confident without being loud.

Voice Attributes

Precise

Say exactly what you mean. No filler words, no ambiguous language, no marketing approximations.

Boundaries
  • Specific over general. "3 workflows deployed" not "workflows updated."
  • Numbers over adjectives. Measurements over feelings.
Anti-patterns
  • Vague reassurance: "Everything looks great!"
  • Weasel words: "Might," "possibly," "up to"

Direct

Lead with the action or the answer. Skip the preamble. Respect the reader's time.

Boundaries
  • Active voice by default. "Deploy the workflow" not "The workflow can be deployed."
  • Front-load the important word in every sentence.
Anti-patterns
  • Burying the lead: "In order to achieve better results, you might want to consider..."
  • Passive deflection: "An error was encountered."

Grounded

Speak from evidence, not aspiration. We describe what things do, not what they could theoretically become.

Boundaries
  • Present tense for current capabilities. Future tense only for roadmap.
  • Technical accuracy over marketing momentum.
Anti-patterns
  • Hype language: "Revolutionary," "game-changing," "powered by magic"
  • Unfounded superlatives: "The most advanced," "the only platform"

Tone Shifts by Context

The voice stays constant. The tone adjusts to context.

Onboarding

Warmer. Allow a moment of welcome. Still direct, but acknowledge the human on the other side. "Welcome. Let's configure your first workspace."

Errors

Calm and specific. Name the problem, state what happened, offer the next step. Never blame the user. "Deployment failed: insufficient permissions on staging-east."

Marketing

Confident, not hyperbolic. Let the product speak. Describe real capabilities with real specificity. Avoid "unlock," "supercharge," "leverage."

Documentation

Clear, concise, scannable. Short paragraphs. Code examples over prose explanations. No opinion unless it prevents an error.

CTA Conventions

Verb-first, action-specific. The label should describe exactly what happens when clicked.

Do

"Deploy workflow"
"Create agent"
"Export as CSV"
"Connect data source"

Don't

"Get started"
"Submit"
"Click here"
"Learn more"

Naming Principles

Do

"Workflow" (clear, functional)
"Agent log" (descriptive)
"Run history" (plain language)
"Deployment target" (technical but precise)

Don't

"TheoFlow™" (jargon for jargon's sake)
"Your AI assistant thinks..." (anthropomorphizing)
"Brainspace" (clever naming over clarity)
"SynapseLink" (branded complexity)

Typography Usage

Source Serif 4 and Inter create editorial tension. Serif signals importance and presence. Sans-serif does the work. Never mix these roles.

The Pairing

Source Serif 4

Use for: Display headings, hero headlines, marketing headlines, empty state headings, page titles on landing pages.

Weight: 400 always. Never bold.

Inter

Use for: Everything else. Body text, UI labels, navigation, form elements, buttons, table headers, subheadings, component text.

Weight: 400 default, 450 for emphasis, 550 for nav active and table headers.

Body Text

Body Large renders at 19px for lead paragraphs and featured content.

19px / Inter / 400 / 1.6

Body Base renders at 17px. This is the default reading size for all body text across the platform.

17px / Inter / 400 / 1.5

14px / Inter / 400 / 1.5

Monospace

JetBrains Mono for code blocks, agent IDs, technical values, API endpoints, and system-generated identifiers. Never for body text or headings.

Tracking Rules

Letter-spacing is size-dependent. Large text gets tight. Small text gets wide. This creates optical consistency.

Size Range Token Value Rationale
48px+ (Display) --tracking-tight -0.025em Large text needs tightening to feel cohesive
28-47px (Heading) --tracking-snug -0.015em Moderate tightening for section headings
17-27px (Body) --tracking-normal 0 Default tracking at reading sizes
14-16px (Small) --tracking-wide 0.01em Slight widening aids readability
13px and below --tracking-wider 0.025em Caps labels and meta text need open tracking

Weight Rules

Prefer weight 400 for all text. Use 450 for subtle emphasis within body text or card titles. Reserve 550 and above for navigation active states, table headers, and strong system labels only. Theorem does not shout.

03

Color Usage

Color is a tool for hierarchy and state, not decoration. The vast majority of any Theorem surface is neutral. Color appears with intention.

When to Use the Accent

The accent color (#b87a4b) has exactly four sanctioned uses.

Primary CTAs

Filled buttons for the primary action on a screen.

Active States

Selected tabs, active navigation items, toggle on-states.

Links

Inline text links and navigation highlights.

Focus Rings

Keyboard focus indicators on all interactive elements.

When Not to Use Color

Do

Use warm grays for borders and dividers.
Use off-white tints for section backgrounds.
Use weight or size changes for heading emphasis.
Use spacing to create visual hierarchy.

Don't

Colored borders on structural containers.
Tinted section backgrounds for visual variety.
Colored heading text for emphasis.
Multiple accent colors for "visual interest."

Functional Colors

Functional colors appear only in their semantic context. They never appear as decoration.

Success

#3a9a6b — Completed actions, successful deployments, passing checks.

Warning

#c48a2c — Approaching limits, degraded performance, requires attention.

Danger

#c44b3f — Errors, failed deployments, destructive actions, system failures.

Info

#4a7fb5 — Neutral informational messages, tips, contextual guidance.

Text Color Hierarchy

Three levels of text color create clear reading priority across every surface.

Primary

Headings, body text, labels, values that must be read first.

--text-primary / gray-900

Secondary

Descriptions, supporting paragraphs, form help text.

--text-secondary / gray-500

Tertiary

Timestamps, placeholder text, disabled labels, meta.

--text-tertiary / gray-400

Light & Dark Themes

Same semantic system, different palette. Both themes maintain identical hierarchy and contrast relationships.

Light Theme

Primary text on warm off-white.

Secondary text for supporting content.

Tertiary text for metadata.

Dark Theme

Primary text on true black.

Secondary text for supporting content.

Tertiary text for metadata.

Card vs Inline Sections

Use Cards When

Content can be acted upon independently.
Items represent discrete objects (agents, workflows).
The surface is elevated from the page (modals, panels).
Content needs visual separation from siblings.

Use Inline Sections When

Content is part of a continuous reading flow.
Items share context and should feel connected.
The hierarchy is established by typography alone.
You need maximum information density.

Motion Guidelines

Animation reinforces spatial relationships and provides feedback. It should feel inevitable, not performative. Every animation must answer: "What does this help the user understand?"

Easing Curves

Four custom curves. Never use CSS presets (ease, ease-in-out). Each curve has a specific character.

Ease Out

cubic-bezier(0.22, 0.61, 0.36, 1)

Fast start, gentle land. Default for most UI transitions.

Ease In-Out

cubic-bezier(0.45, 0.05, 0.55, 0.95)

Symmetric. For state changes that feel deliberate.

Spring

cubic-bezier(0.34, 1.56, 0.64, 1)

Overshoots slightly. For playful micro-interactions and toggles.

Smooth

cubic-bezier(0.52, 0.01, 0.26, 1)

Heavy, deliberate. For page transitions and large reveals.

Duration Rules

160ms

Micro-interactions

Hover states, button press, toggle, focus rings, tooltip show/hide.

280ms

Transitions

Panel open/close, tab switching, dropdown expand, card state changes.

450ms

Reveals

Content entering viewport, modal appearance, page section fade-in.

Interaction Signatures

Hover Scale

scale(1.015)

Press Scale

scale(0.98)

Stagger Delay

60ms between items

Reduced Motion

Reduced motion is a parallel design track, not disabled animation. When prefers-reduced-motion: reduce is active, all durations go to 0ms, scale transforms are disabled, and stagger delays are removed. The layout and states remain identical — only the interpolation is removed.

Motion Boundaries

Do

Animate content entering or leaving the viewport.
Animate state feedback (hover, press, focus).
Animate sequential item appearance with stagger.
Use spring easing for toggle switches.

Don't

Animate structural layout changes (grid reflow).
Add loading spinners longer than 700ms without context.
Use bounce or elastic easing on data-heavy views.
Animate color changes on large surface areas.

Accessibility

Accessibility is a design constraint, not a compliance checkbox. Every decision in this system accounts for the widest possible range of human interaction.

Contrast Ratios

All semantic color pairings are tested against WCAG 2.1 criteria. AA is the minimum. AAA is the target for body text.

Pairing Ratio AA (4.5:1) AAA (7:1)
Primary text on surface 15.2:1 Pass Pass
Secondary text on surface 5.8:1 Pass Fail
Tertiary text on surface 4.6:1 Pass Fail
Accent on surface 5.1:1 Pass Fail
White on accent button 4.9:1 Pass Fail
Success on surface 4.7:1 Pass Fail
Danger on surface 5.3:1 Pass Fail

Interaction Accessibility

Focus Management

All interactive elements show a visible focus ring using the accent color (#b87a4b) with a 2px solid outline and 2px offset. Focus rings are never suppressed. The outline uses :focus-visible to avoid showing on mouse clicks.

Keyboard Navigation

All interactive components are keyboard navigable. Tab order follows visual reading order. Modals trap focus. Escape closes overlays. Arrow keys navigate within composite widgets (tabs, menus, radio groups).

Screen Readers

Use aria-live="polite" for status changes (deployments, workflow completions). Provide meaningful alt text for all images. Icon-only buttons require aria-label. Form inputs require associated labels.

Color Independence

Never convey meaning through color alone. Every status indicator pairs color with an icon or text label. Error states use both red color and an error icon. Success uses both green and a check icon. Charts use patterns in addition to colors.

07

Visual Assets Direction

Every visual element in Theorem is intentional. Photography is editorial. Illustration is structural. Icons are precise. Graphic devices are restrained.

Photography

High contrast, desaturated, warm tones. Shot with intention — not stock. Subjects should feel candid but composed. Avoid overly staged environments. Prefer detail shots over wide establishing shots.

High contrast Desaturated Warm tones

Illustration

Geometric, restrained palette limited to grays plus the accent color. Line-based, not filled. Convey structure and relationships, not emotion. Avoid gradients within illustrations.

Geometric Line-based Gray + accent

Iconography

20px default size on a square grid. 1.5px stroke weight. No fills — outline only. Rounded caps and joins. Optically aligned within the grid. Every icon must be legible at 16px.
