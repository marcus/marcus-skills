Guidelines

Principles, patterns, and rules for building coherent interfaces.

Principles

Design Principles

1 Warm precision

We combine technical confidence with human warmth. Every sharp edge is balanced by an organic texture. Precision without coldness.

DO
Use amber accents to add warmth to data-heavy screens
DON'T
Use cool blue-gray as the dominant surface color
2 Purposeful restraint

Every element earns its place. White space is a feature, not a gap. Density is a choice, not a default.

DO
Let content breathe with generous spacing between sections
DON'T
Fill every available pixel with UI elements
3 Agent-first clarity

Autonomous systems need transparent interfaces. Status, progress, and decision-making should be visible and understandable at a glance.

DO
Show agent status with clear indicators and human-readable summaries
DON'T
Hide system behavior behind abstract progress bars
4 Editorial confidence

Our typography and layout choices convey authority. We present information with the care of a well-edited publication.

DO
Use the serif display face for key statements and page titles
DON'T
Mix more than 2 typefaces on a single screen

Tokens

Token Usage

The three-layer model ensures design decisions propagate predictably from raw values to finished components.

P
Primitive
Brand-agnostic raw values. Colors, sizes, durations.
S
Semantic
Meaningful roles. The primary API for consuming the system.
C
Component
Scoped overrides when semantic tokens can't express the need.

Naming convention

Layer Pattern Example
Primitive --p-{category}-{value} --p-amber-600
Semantic --color-{role}-{variant} --color-accent-default
Component --{component}-{property} --btn-height-md

Rules

  • Never use primitive tokens in components directly
  • Semantic tokens are the primary API for consuming the design system
  • Component tokens exist only when a component needs values that can't be expressed semantically

Status color rules

Never use color alone to convey meaning — always pair with text, icon, or shape.
Status backgrounds (subtle variants) are for containers; status defaults are for indicators and badges.
The amber accent should be reserved for primary actions and brand moments — don't use it for warnings in the same context.

Motion

Motion Guidelines

When to animate

  • State changes (hover, focus, active, disabled)
  • Entrances and exits (modals, dropdowns, toasts)
  • Feedback (success confirmations, error shakes)

When NOT to animate

  • Content that's already visible on screen
  • Decorative motion with no functional purpose
  • Anything on page load that delays comprehension

Duration guidance

Micro-interactions
120ms
Component transitions
200ms
Page transitions
350ms
Dramatic reveals
500ms+

Reduced motion

Always provide a parallel track — use opacity-only transitions instead of transforms when prefers-reduced-motion is active. Motion should degrade gracefully, never disappear entirely.

Accessibility

Accessibility

Aa
Contrast
Minimum 4.5:1 for normal text, 3:1 for large text (18px+ or 14px bold). All semantic color pairings are validated against WCAG AA.
Focus indicators
2px amber ring with 2px offset on all interactive elements. Focus must be visible in both light and dark themes.
Keyboard navigation
Tab for focus, Enter/Space for activation, Escape for dismissal. All interactive patterns must be operable without a pointer device.
Screen reader
All interactive elements must have accessible names. Status changes announced via aria-live. Decorative elements hidden with aria-hidden="true".
Color independence
Status indicators must include icon or text label alongside color. Never rely on color as the sole differentiator.
