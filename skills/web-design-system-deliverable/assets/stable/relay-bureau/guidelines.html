Usage rules

A strong system is mostly made of boundaries.

Relay Bureau’s differentiation comes from a few disciplined choices repeated consistently: structural rails instead of card grids, type contrast instead of decorative noise, motion as confirmation, and copy that sounds like a composed producer rather than a chatbot.

What to protect
  • Do not flatten the type hierarchy. Large vs. small contrast is doing real brand work.
  • Do not fill neutral surfaces with cobalt. Accent is a signal, not a backdrop.
  • Do not replace rails with stacked cards. Layout is part of the identity.
  • Do not overuse serif. It is the release valve, not the default voice.
Voice and tone

Direct, literate, and slightly stern.

The product speaks like a senior producer. It is not robotic, but it also never performs friendliness when clarity is needed. Tone shifts by context, but the underlying voice stays composed.

Voice attributes

  • Precise: describe what changed, who owns it, and what happens next.
  • Editorial: prefer framing that reveals judgment, not just status.
  • Controlled: avoid exclamation points, hype words, and synthetic encouragement.

CTA pattern

  • Primary: verb first. Example: “Release summary”, “Open review”, “Route for approval”.
  • Secondary: noun or short verb phrase. Example: “Revision notes”, “Assign owner”.
  • Danger: full phrase, never a single dramatic word. Example: “Archive this route”.

Error and empty-state copy

  • Error pattern: state the mismatch, then give one next step.
  • Empty pattern: explain why the lane is empty and whether that is desirable.
  • Never use motivational filler like “You’re all caught up!” unless the product genuinely benefits from warmth.
Composition rules

Use structure before ornament.

The pages are designed to work like production layouts. Long-form explanation goes wide. Data and options live in rails. Cards are reserved for repeated items or contained states, never for the whole page architecture.

Do
  • Build pages with a rail, a stage, and an inspector when complexity is high.
  • Switch density by context: relaxed for overview, standard for narrative work, compact for tables.
  • Pair big statements with tight metadata rows so the interface feels authored.
Don’t
  • Turn every section into a rounded card grid.
  • Use one radius everywhere. Relay Bureau depends on structural, control, and signature shapes staying distinct.
  • Introduce dark mode as a separate art direction. It should feel like the same room with the lights lowered.
Visual assets

The asset system is architectural, not illustrative.

The design references point toward strong product staging, custom-type confidence, and art direction that can live in both marketing and software. Relay Bureau keeps that but refuses mascot-ized AI imagery.

Photography: directional light, desks, prototypes, materials, and real devices. Cropped for tension, not softness.
Graphic devices: route lines, capsule counters, cropped bars, and thin separators that suggest flow without becoming diagrams.
Iconography: low-stroke geometric shapes that can hold up inside compact rails, dense tables, and command bars.
Illustration: abstract systems and editorial notation only. No floating robots, glowing brains, or generic holograms.
Responsive strategy

Collapse intelligently. Don’t just stack.

The product shell is intended to reflow from three columns to a single stage with collapsible sections. Sidebars become top modules. Inspector content becomes inline detail blocks. Marketing sections keep their large type, but utility metadata compresses early.

Breakpoints

  • Wide: 1152px and up for rail + stage + inspector.
  • Medium: 768px to 1151px for stage-first with stacked support panels.
  • Compact: below 768px with chip rows wrapping and forms expanding full width.

Touch targets

  • Primary controls keep 48px minimum tap height.
  • Chip groups wrap rather than shrinking below readability.
  • Dense tables switch to stacked rows with strong field labels when space collapses.

Container logic

  • Hero statements scale fluidly.
  • Body copy stays fixed for reading stability.
  • Rails can disappear; hierarchy cannot.
Accessibility audit

AA contrast, visible focus, keyboard-first movement, reduced motion by default path.

Accessibility is treated here as documentation, not a single checkbox. The system assumes asynchronous work, frequent route changes, and review flows that need audible and visual confirmation.

Contrast

Paper + ink, cobalt + paper, moss + paper, and inverse surfaces all clear AA for primary text. Accent tints are used for backgrounds only when paired with dark text.

Keyboard map

Command row, filters, sidebar items, tables, and drawers are reachable in source order. Opening overlays moves focus in; closing returns focus to the trigger.

Screen reader behavior

Async save, publish, and routing updates should announce through polite live regions. Error summaries should link to the field with the issue.

Reduced motion

Staggers, shimmer, and slide-in choreography all degrade to instant state change when users opt out of animation.

Governance and handoff

Protect the thesis while leaving room for extension.

New components should inherit the system’s structural rules before they inherit its look. If a new surface breaks the rail-stage-rail logic, it needs a stronger argument than “this is how the framework generated it.”

Extension rules

  • New components must map to existing shape roles before introducing new radii.
  • Any new accent color must attach to a semantic state, not a campaign.
  • Visual experiments belong in proof pages before entering the component library.

Token integration

  • Source of truth: `tokens.json` plus CSS custom properties in `styles/tokens.css`.
  • Suggested pipeline: Style Dictionary to CSS, JSON, iOS, Android.
  • Figma setup: primitives, semantics, components, and theme modes as separate variable groups.

Handoff checklist

  • Ship interactive states and copy rules together.
  • Document which proof pages validate new components.
  • Keep screenshots of light and dark mode in the package for fast review.

Relay Bureau is opinionated on purpose. If a future extension starts to look generic, the answer is usually to restore the layout and type logic, not to add more effects.
